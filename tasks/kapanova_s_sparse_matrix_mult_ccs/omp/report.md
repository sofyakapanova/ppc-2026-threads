## 1. Контекст
В OpenMP-версию из SEQ перенесён внешний цикл по столбцам матрицы `B`. Вместо последовательного прохода по столбцам они распределяются между потоками. Логика обработки одного столбца оставлена без изменений и вынесена в `ProcessColumn`. Каждый поток использует собственные массивы для накопления результатов вместо одного общего `accum`.

## 2. Постановка задачи
Задача: перемножить две разреженные матрицы `A` и `B` с элементами типа double в формате CCS с использованием OMP. Ограничения, крайние случаи и критерии корректности совпадают с `seq/report.md`.
Последовательная версия используется как эталон корректности и baseline для расчёта ускорения.
Цели:
- уменьшить время выполнения по сравнению с SEQ
- исследовать масштабируемость алгоритма

## 3. Базовый алгоритм
Последовательный алгоритм (см. `seq/report.md`, раздел 3) проходит по столбцам матрицы `B`. Для каждого столбца `j` перебираются его ненулевые элементы. Для каждого такого элемента с индексом строки `k` и значением `b_val` обрабатывается столбец `k` матрицы `A`: значения `a_val` умножаются на `b_val` и накапливаются во временном векторе `accum` по индексу строки `i`. Затем измененные строки сортируются, ненулевые значения и их индексы записываются в результирующую матрицу `C` в формате CCS, а `accum` сбрасывается для следующего столбца.

## 4. Схема распараллеливания
**Какая область параллелится:**
Внешний цикл по столбцам `j = 0 .. B.cols-1`. Каждый поток обрабатывает свой набор столбцов независимо.

**Переменные — `shared`:**
- `a`, `b` — входные матрицы 
- `c` — выходная матрица 
- `thread_accum`, `thread_row_mask`, `thread_active_rows`, `thread_col_rows`, `thread_col_vals` — общие структуры, индексируемые по номеру потока `[tid]`.

**Переменные — `private`:**
- `j` — счётчик цикла 
- Внутри `ProcessColumn`: локальные переменные каждого потока.

**Почему не используются `reduction`, `atomic`, `critical`:**

Каждый поток пишет результаты только в свои массивы по индексу `tid` (номер потока):
- `thread_accum[tid][i]` — накопление суммы,
- `thread_col_rows[tid][j]` — индексы строк для столбца `j`,
- `thread_col_vals[tid][j]` — значения для столбца `j`.

Два разных потока никогда не обращаются к одной и той же ячейке. 

`reduction` не подходит, потому что результат — не скаляр, а структура CCS с индексами и значениями. `atomic` и `critical` не нужны, так как общие данные только читаются, а запись идёт в приватные области.

**Какой `schedule` выбран и почему:**
Используется `schedule(dynamic)`. Столбцы матрицы `B` могут содержать разное количество ненулевых элементов, поэтому вычислительная нагрузка на столбец неравномерна. `dynamic` позволяет потокам брать новые столбцы по мере завершения предыдущих, балансируя загрузку.

**Неявные барьеры:**
- Конец первого `#pragma omp parallel` — барьер, гарантирующий, что все потоки инициализировали свои локальные массивы.
- Конец `#pragma omp parallel for` — барьер перед переходом к слиянию результатов. Необходим так как `ComputeColumnSizes` и `MergeThreadResults` работают последовательно и требуют полных данных от всех потоков.

## 5. Детали реализации
Файлы: omp/include/ops_omp.hpp, omp/src/ops_omp.cpp

**Изменения относительно SEQ:**
- Добавлен параллельный регион для инициализации приватных массивов каждого потока (`thread_accum`, `thread_row_mask`, `thread_col_rows`, `thread_col_vals`).
- Внешний цикл по столбцам `B` распараллелен через `#pragma omp parallel for schedule(dynamic)`.
- Логика обработки одного столбца вынесена в отдельный метод `ProcessColumn`. Каждый поток вызывает `omp_get_thread_num()` и работает только со своими массивами по индексу `tid`.
- После параллельного региона последовательно выполняются:
  - `ComputeColumnSizes` — суммирует размеры столбцов от всех потоков;
  - заполнение `c.col_ptrs` и выделение памяти под `c.values` и `c.row_indices`;
  - `MergeThreadResults` — копирует данные из приватных массивов потоков в `c`.

**Как устранены риски гонок:**
Каждый поток пишет строго в свои ячейки: `thread_accum[tid][i]`, `thread_col_rows[tid][j]`, `thread_col_vals[tid][j]`. Индекс `tid` уникален для потока. Общие структуры `thread_*` индексируются по `tid` — два потока никогда не пишут в одну ячейку. 

**Ключевой фрагмент — основная директива:**
```cpp
// omp/src/ops_omp.cpp — распараллеливание внешнего цикла
#pragma omp parallel for schedule(dynamic) default(none) \
    shared(a, b, c, thread_accum, thread_row_mask, thread_active_rows, thread_col_rows, thread_col_vals)
  for (size_t j = 0; j < c.cols; ++j) {
    ProcessColumn(static_cast<int>(j), a, b, thread_accum, thread_row_mask, thread_active_rows,
                  thread_col_rows, thread_col_vals);
  }
```
schedule(dynamic) выбран из-за неравномерной загрузки столбцов. default(none) требует явно перечислить все shared-переменные — это страхует от случайного превращения приватной переменной в общую и наоборот. Неявный барьер в конце цикла гарантирует, что все потоки завершили обработку столбцов перед слиянием результатов.

## 6. Проверка корректности
Корректность проверяется теми же пятью функциональными тестами, что и SEQ (см. `seq/report.md`, раздел 5). Результат OMP-версии поэлементно сравнивается с эталоном с допуском eps = 1e-10: при OMP_NUM_THREADS=1,2,4,8 все тесты проходят - результат стабилен и не зависит от числа потоков.

## 8. Результаты

Тесты производительности запущены в обоих режимах (`task_run` и `pipeline`), все выполнилось в пределах лимитов. Ускорение считается по `task_run`.

| Version | Threads | Время task (с)| Ускорение | Эффективность |
|---------|---------|---------------|-----------|---------------|
| SEQ     | 1       | 2.289         | 1.00      | 100%          |
| OMP     | 2       | 1.783         | 1.28      | 64%           |
| OMP     | 4       | 1.052         | 2.18      | 54%           |
| OMP     | 8       | 1.040         | 2.20      | 28%           |

Время в режиме `pipeline`: SEQ — 2.326 с, OMP (2/4/8) — 1.703 / 1.160 / 0.993 с.

Ускорение считается как T_seq / T_omp. Эффективность = ускорение / число_потоков × 100%.

**О масштабируемости:**

При переходе от 1 к 4 потокам ускорение растёт (1.28 → 2.18), но при 8 потоках почти не меняется (2.18 → 2.20). Эффективность падает с 64% на 2 потоках до 28% на 8. Это означает, что после 4 потоков алгоритм упирается в ограничения:
1. **Нерегулярный доступ к памяти.** Индекс `i = a.row_indices[zc]` меняется непредсказуемо, из-за чего данные не попадают в кэш. При увеличении числа потоков растёт конкуренция за доступ к памяти, но характер доступа остаётся прежним, поэтому проблема не устраняется.
2. **Последовательное слияние результатов.** После параллельного региона `MergeThreadResults` работает в одном потоке, и это время не уменьшается при добавлении потоков.
3. **Особенности Apple M1.** Чип имеет 4 производительных ядра и 4 энергоэффективных. Основной прирост дают первые 4 потока на производительных ядрах. Потоки 5–8 работают на других и почти не сокращают время.

## 9. Выводы

OpenMP дал выигрыш по сравнению с SEQ: максимальное ускорение 2.20× на 8 потоках. Максимальная эффективность (64%) получена при 2 потоках — накладные расходы минимальны, оба потока работают на производительных ядрах. 

Ограничения масштабируемости:
- нерегулярный доступ к памяти (данные не попадают в кэш во внутреннем цикле);
- последовательный этап слияния результатов;
- архитектура процессора (после 4 потоков задействуются медленные ядра).

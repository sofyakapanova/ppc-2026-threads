## 1. Контекст

STL-версия (`std::thread`) реализует ручное управление потоками. В отличие от OpenMP, где распределение работы и синхронизация скрыты в директивах, здесь всё делается явно: деление данных, создание потоков, `join`. Это позволяет увидеть полную картину накладных расходов и проверить, насколько эффективна изоляция данных без автоматических механизмов синхронизации.

В STL-версию из SEQ перенесён внешний цикл по столбцам матрицы `B`. Столбцы делятся на непрерывные диапазоны, каждый поток обрабатывает свой диапазон независимо. Логика обработки одного столбца сохранена. Для устранения гонок каждый поток работает с собственным экземпляром `ThreadLocalData`.

## 2. Постановка задачи
Задача: перемножить две разреженные матрицы `A` и `B` с элементами типа double в формате CCS с использованием std::thread (STL).
Ограничения, крайние случаи и критерии корректности совпадают с `seq/report.md`. 
Последовательная версия используется как эталон корректности и baseline для расчёта ускорения.

Цели:
- уменьшить время выполнения по сравнению с SEQ
- исследовать масштабируемость алгоритма

## 3. Базовый алгоритм
Последовательный алгоритм (см. `seq/report.md`, раздел 3) проходит по столбцам матрицы `B`. Для каждого столбца `j` перебираются его ненулевые элементы. Для каждого такого элемента с индексом строки `k` и значением `b_val` обрабатывается столбец `k` матрицы `A`: значения `a_val` умножаются на `b_val` и накапливаются во временном векторе `accum` по индексу строки `i`. Затем измененные строки сортируются, ненулевые значения и их индексы записываются в результирующую матрицу `C` в формате CCS, а `accum` сбрасывается для следующего столбца.

## 4. Схема распараллеливания
**Как данные делятся на диапазоны:**
Столбцы результирующей матрицы `C` (индексы `0 .. B.cols-1`) делятся на равные непрерывные блоки. Размер блока: `(cols + num_threads - 1) / num_threads`. Поток `t` обрабатывает столбцы с `t * chunk` до `min((t+1) * chunk, cols)`.

**Что делает каждый поток:**
Выполняет функцию `WorkerTask`. Cоздаёт свой экземпляр `ThreadLocalData` (векторы `accum`, `mask`, `active`). Для каждого столбца из своего диапазона вызывает `ProcessColumn`. Результаты записывает в общие векторы `temp_rows[j]` и `temp_vals[j]`.

**Где хранятся локальные результаты:**
- Промежуточные (внутри потока) — в `ThreadLocalData::accum`, `mask`, `active`.
- Итоговые для столбца — в общих `temp_rows[j]` и `temp_vals[j]`. Поскольку диапазоны столбцов у потоков не пересекаются, разные потоки пишут в разные `j` — гонок нет.

**Какая синхронизация используется:**
Синхронизация не используется — `mutex`, `atomic`, `condition_variable` отсутствуют. Потоки работают с непересекающимися диапазонами столбцов и собственными локальными данными, поэтому гонки невозможны. Единственная точка синхронизации — `join()`, гарантирует завершение всех потоков перед слиянием результатов.

## 5. Детали реализации
Файлы: stl/include/ops_stl.hpp, stl/src/ops_stl.cpp

**Изменения относительно SEQ:**
- Добавлена структура `ThreadLocalData` — приватные для потока `accum`, `mask`, `active`. Метод `ClearActive()` сбрасывает состояние перед обработкой следующего столбца.
- Функция `ProcessColumn` — логика обработки одного столбца, аналогичная SEQ, но работающая с `ThreadLocalData`.
- Функция `WorkerTask` — создаёт `ThreadLocalData` и обрабатывает назначенный диапазон столбцов.
- В `RunImpl`: определение числа потоков через `std::thread::hardware_concurrency()`, деление столбцов на блоки, создание потоков, `join`, формирование `col_ptrs` и копирование результатов в `c`.

**Карта поток → диапазон:**
| Поток | Столбцы                             |
|-------|-------------------------------------|
| 0     | `[0, chunk)`                        |
| 1     | `[chunk, 2*chunk)`                  |
| ...   | ...                                 |
| t     | `[t*chunk, min((t+1)*chunk, cols))` |

**Схема объединения локальных результатов:**
После `join` главный поток проходит по всем столбцам `j`, вычисляет `col_ptrs[j]` как накопительную сумму размеров `temp_rows[j]`, затем копирует индексы и значения из `temp_rows[j]` и `temp_vals[j]` в `c.row_indices` и `c.values`.

**Ключевой фрагмент — создание и запуск потоков:**
```cpp
// stl/src/ops_stl.cpp — создание потоков
for (unsigned int thread_id = 0; thread_id < num_threads; ++thread_id) {
    size_t start = thread_id * chunk;
    size_t end = std::min(start + chunk, c.cols);
    if (start >= c.cols) break;

    threads.emplace_back(WorkerTask, std::cref(a), std::cref(b),
                         static_cast<int>(start), static_cast<int>(end),
                         std::ref(temp_rows), std::ref(temp_vals));
}

for (auto &th : threads) {
    if (th.joinable()) th.join();
}
```
Потоки создаются в первом цикле и сразу начинают работу. join() вызывается во втором цикле — все потоки работают параллельно. std::cref и std::ref используются для передачи матриц и результирующих векторов по ссылке без копирования.

## 6. Проверка корректности
Корректность проверяется теми же пятью функциональными тестами, что и SEQ (см. `seq/report.md`, раздел 5). Результат STL-версии поэлементно сравнивается с эталоном с допуском eps = 1e-10: при PPC_NUM_THREADS=1,2,4,8 все тесты проходят - результат стабилен и не зависит от числа потоков.

Гонки исключены: каждый поток работает с непересекающимся диапазоном столбцов и собственным экземпляром ThreadLocalData.

## 8. Результаты
Тесты производительности запущены в обоих режимах (`task_run` и `pipeline`), все выполнилось в пределах лимитов. Ускорение считается по `task_run`.

| Version | Threads | Время task (с)| Ускорение | Эффективность |
|---------|---------|---------------|-----------|---------------|
| SEQ     | 1       | 2.289         | 1.00      | 100%          |
| STL     | 2       | 0.750         | 3.05      | 153%          |
| STL     | 4       | 0.773         | 2.96      | 74%           |
| STL     | 8       | 0.708         | 3.23      | 40%           |

Время в режиме `pipeline`: STL (2/4/8) — 0.751 / 0.716 / 0.714 с.

Ускорение считается как T_seq / T_stl. Эффективность = ускорение / число_потоков × 100%.

При 2 потоках наблюдается ускорение 3.05×, эффективность 153% => два потока эффективнее используют кэш, чем один. При 4 и 8 потоках время почти не меняется (0.750 → 0.773 → 0.708). Это означает, что после 4 потоков алгоритм упирается в ограничения:
**Ограничения масштабируемости:**
1. **Нерегулярный доступ к памяти.** Индекс `i = a.row_indices[zc]` меняется непредсказуемо, из-за чего данные не попадают в кэш. При увеличении числа потоков растёт конкуренция за доступ к памяти, но характер доступа остаётся прежним, поэтому проблема не устраняется.
2. **Последовательное слияние результатов.** После join главный поток последовательно копирует результаты из `temp_rows` и `temp_vals` в `c` — это время не уменьшается при добавлении потоков.
3. **Особенности Apple M1.** Чип имеет 4 производительных ядра и 4 энергоэффективных. Основной прирост дают первые 4 потока на производительных ядрах. Потоки 5–8 работают на других и почти не сокращают время.

Накладные расходы на создание потоков и `join` пренебрежимо малы по сравнению с временем умножения матриц.

## 9. Выводы

STL-версия с ручным управлением потоками дала максимальное ускорение 3.23× на 8 потоках по сравнению с SEQ. На 2 потоках достигнуто ускорение 3.05× за счёт использования кэша при работе двух потоков на производительных ядрах.

Статическое разбиение на непрерывные диапазоны в паре с переиспользованием `ThreadLocalData` внутри потока оказалось эффективным. Гонки исключены без какой-либо синхронизации за счёт полной изоляции данных потоков.

Ограничения масштабируемости:
- нерегулярный доступ к памяти (данные не попадают в кэш во внутреннем цикле);
- последовательный этап слияния результатов;
- архитектура процессора (после 4 потоков задействуются медленные ядра).

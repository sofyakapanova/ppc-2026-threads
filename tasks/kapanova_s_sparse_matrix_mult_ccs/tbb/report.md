## 1. Контекст
TBB-версия использует Intel oneTBB для параллельного умножения разреженных матриц. В отличие от ручного управления потоками в STL и директив в OpenMP, TBB предоставляет высокоуровневые примитивы: `parallel_for` автоматически распределяет итерации по потокам, а `enumerable_thread_specific` обеспечивает каждому потоку собственный экземпляр локальных данных без явного создания массивов по индексу `tid`.

## 2. Постановка задачи
Задача: перемножить две разреженные матрицы `A` и `B` с элементами типа double в формате CCS с использованием oneTBB. Ограничения, крайние случаи и критерии корректности совпадают с `seq/report.md`. Последовательная версия используется как эталон корректности и baseline для расчёта ускорения.

**Цели:**
- Уменьшить время выполнения по сравнению с SEQ.
- Исследовать масштабируемость алгоритма при использовании TBB.

## 3. Базовый алгоритм
Последовательный алгоритм (см. `seq/report.md`, раздел 3) проходит по столбцам матрицы `B`. Для каждого столбца `j` перебираются его ненулевые элементы. Для каждого такого элемента с индексом строки `k` и значением `b_val` обрабатывается столбец `k` матрицы `A`: значения `a_val` умножаются на `b_val` и накапливаются во временном векторе `accum` по индексу строки `i`. Затем измененные строки сортируются, ненулевые значения и их индексы записываются в результирующую матрицу `C` в формате CCS, а `accum` сбрасывается для следующего столбца.


## 4. Схема распараллеливания
**Какой примитив выбран:**
`tbb::parallel_for` — параллельное выполнение цикла по столбцам. В отличие от `parallel_reduce`, здесь не требуется операция редукции с комбинацией результатов — каждый поток пишет в свои массивы, а слияние делается после.

**Как определяется диапазон работы:**
`tbb::blocked_range<size_t>(0, c.cols, 256)` — столбцы от 0 до `c.cols` делятся на блоки. Параметр `256` — grainsize (минимальный размер блока в итерациях).

**Какой grainsize используется и почему:**
`grainsize = 256`. Это компромисс: слишком маленький grainsize (например, 1) создаст много мелких задач и увеличит накладные расходы на планирование; слишком большой — ухудшит балансировку. Значение 256 выбрано эмпирически: при 10000 столбцах это даёт около 40 блоков, что достаточно для балансировки между потоками без избыточного дробления.

**Какой partitioner используется:**
Явно не указан — TBB использует `auto_partitioner` по умолчанию. Он автоматически выбирает между `simple_partitioner` (равные блоки) и `affinity_partitioner` (учёт кэша) в зависимости от размера диапазона и grainsize.

**Как ограничивалась конкуренция:**
Конкуренция не ограничивается явно. `tbb::global_control` не используется — TBB самостоятельно определяет число рабочих потоков на основе `std::thread::hardware_concurrency()`.

**Локальные данные:**
`tbb::enumerable_thread_specific<ThreadLocalBuffers>` — каждый поток автоматически получает свой экземпляр `ThreadLocalBuffers` (accum, mask, active_rows, col_rows, col_vals). Обращение через `tls_data.local()`. Это аналог `thread_local`, но управляется TBB.

## 5. Детали реализации
Файлы: tbb/include/ops_tbb.hpp, tbb/src/ops_tbb.cpp

**Изменения относительно SEQ:**
- Добавлена структура `ThreadLocalBuffers` — приватные для потока accum, mask, active_rows, col_rows, col_vals.
- Функция `ProcessColumnRange` — обрабатывает диапазон столбцов `[range.begin(), range.end())`, работает с локальными данными потока.
- В `RunImpl`: создание `tls_data` через `enumerable_thread_specific`, параллельный `parallel_for` с `blocked_range`, последовательные `MergeColumnSizes` и `MergeResults`.
- `MergeColumnSizes` — обходит все экземпляры `tls_data` и суммирует размеры столбцов.
- `MergeResults` — копирует индексы и значения из локальных данных в `c`.

**Локальные аккумуляторы:**
Каждый поток работает с собственным экземпляром `ThreadLocalBuffers`, полученным через `tls_data.local()`. Запись в общие структуры не происходит до последовательного слияния. 

**Объединение результатов:**
После `parallel_for` выполняется два последовательных прохода:
1. `MergeColumnSizes` — обходит все экземпляры `tls_data`, суммирует размеры `col_rows[j].size()` в `col_sizes[j]`.
2. `MergeResults` — копирует индексы и значения из каждого экземпляра `tls_data` в итоговую матрицу `c`.

**Работа с памятью:**
`enumerable_thread_specific` создаёт ровно столько экземпляров `ThreadLocalBuffers`, сколько потоков реально участвует в `parallel_for`. Каждый экземпляр содержит векторы суммарным размером `O(rows + cols)`. Специальные аллокаторы (`cache_aligned_allocator`) не используются, так как потоки не разделяют одни и те же ячейки — false sharing не возникает.

**Ключевой фрагмент:**
```cpp
// tbb/src/ops_tbb.cpp — параллельный цикл
tbb::parallel_for(tbb::blocked_range<size_t>(0, c.cols, 256),
  [&](const tbb::blocked_range<size_t> &range) {
    auto &local = tls_data.local();
    ProcessColumnRange(a, b, local.col_rows, local.col_vals,
                       local.accum, local.mask, local.active_rows, range);
  });
```
blocked_range делит столбцы на блоки размером не менее 256. tls_data.local() возвращает приватные данные текущего потока. После parallel_for выполняется последовательное слияние результатов.

## 6. Проверка корректности
Корректность проверяется теми же пятью функциональными тестами, что и SEQ (см. `seq/report.md`, раздел 5). Результат TBB-версии поэлементно сравнивается с эталоном с допуском eps = 1e-10: при PPC_NUM_THREADS=1,2,4,8 все тесты проходят - результат стабилен и не зависит от числа потоков.

Гонки исключены: каждый поток работает с непересекающимся диапазоном столбцов и собственным экземпляром ThreadLocalBuffers

## 8. Результаты

Тесты производительности запущены в обоих режимах (`task_run` и `pipeline`), все выполнилось в пределах лимитов. Ускорение считается по `task_run`.

| Backend | Threads | Время task (с) | Ускорение | Эффективность |
|---------|:-------:|:-------------:|:---------:|:-------------:|
| SEQ     | 1       | 2.289         | 1.00      | 100%          |
| TBB     | 2       | 0.738         | 3.10      | 155%          |
| TBB     | 4       | 0.413         | 5.54      | 139%          |
| TBB     | 8       | 0.326         | 7.02      | 88%           |

Время в режиме `pipeline`: TBB (2/4/8) — 0.747 / 0.410 / 0.314 с.

Ускорение считается как T_seq / T_tbb. Эффективность = ускорение / число_потоков × 100%.

**Масштабируемость:**

TBB показывает хорошую масштабируемость: ускорение растёт с ростом числа потоков: 3.10× (2 потока) → 5.54× (4 потока) → 7.02× (8 потоков). Эффективность остаётся высокой даже на 8 потоках (88%). Такие ускорения объясняются лучшим использованием кэша при параллелизации.

**balancing и overhead:**

`auto_partitioner` с `grainsize = 256` обеспечил хороший баланс загрузки. В отличие от STL со статическим разбиением, TBB может динамически перераспределять работу. Накладные расходы `enumerable_thread_specific` и планирования задач окупаются за счёт лучшей балансировки.

## 9. Выводы

TBB-версия показала хорошую масштабируемость: ускорение растёт с ростом числа потоков. Максимальное ускорение 7.02× на 8 потоках.

Преимущества TBB в данной задаче:
- `auto_partitioner` эффективнее распределяет столбцы, чем статическое разбиение в STL и `schedule(dynamic)` в OMP;
- `enumerable_thread_specific` автоматически управляет приватными данными потоков;
- grainsize = 256 обеспечивает компромисс между балансировкой и накладными расходами.

Ограничения масштабируемости:
- нерегулярный доступ к памяти (данные не попадают в кэш во внутреннем цикле);
- последовательный этап слияния результатов;
- архитектура процессора (после 4 потоков задействуются медленные ядра).
Гонки исключены за счёт изоляции данных через `enumerable_thread_specific`.
